# cap-js-plugin-template

> A TypeScript template for building SAP CAP CDS plugins

## Overview

This is a starter template for creating [SAP CAP](https://cap.cloud.sap/) plugins using TypeScript. It follows the `cds-plugin.js` auto-discovery convention and provides a well-structured foundation with build tooling, testing, and CI-ready configuration.

Use this template to build your own CAP plugin — just clone, rename, and implement your logic.

## Getting Started

### 1. Clone & Rename

```bash
# Clone the template
git clone https://github.com/YOUR_ORG/cap-js-plugin-template.git my-cap-plugin
cd my-cap-plugin

# Update package.json name, description, repository URLs
```

### 2. Install Dependencies

```bash
npm install
```

### 3. Build

```bash
npm run build
```

### 4. Run Tests

```bash
npm test
```

## How CAP Plugins Work

CAP automatically loads any npm dependency that has a `cds-plugin.js` file in its package root. This file is the entrypoint where you hook into CAP lifecycle events.

### Lifecycle Hooks

| Hook                         | When it fires                  | Use case                           |
| ---------------------------- | ------------------------------ | ---------------------------------- |
| `cds.on('loaded', model)`    | CDS model is loaded            | Modify/extend the model            |
| `cds.on('connect', service)` | A service connects             | Wrap RemoteService methods         |
| `cds.on('serving', service)` | An app service is being served | Add handlers to ApplicationService |
| `cds.once('served', ())`     | All services are served        | Post-startup logic                 |

### Service Types

| Class                    | Description                           |
| ------------------------ | ------------------------------------- |
| `cds.ApplicationService` | Services defined in your srv/ folder  |
| `cds.RemoteService`      | External/remote service connections   |
| `cds.MessagingService`   | Messaging services (Event Mesh, etc.) |
| `cds.DatabaseService`    | Database services                     |

## Customization Guide

### 1. Define Your Config (`src/config.ts`)

Update the `PluginConfig` interface and `DEFAULTS` with your plugin's settings. Replace the `myPlugin` config key with your own:

```typescript
export interface PluginConfig {
  enabled: boolean;
  myOption: string;
}

export const DEFAULTS: PluginConfig = {
  enabled: true,
  myOption: 'default-value',
};
```

Users configure it in their `package.json` or `.cdsrc.json`:

```json
{
  "cds": {
    "myPlugin": {
      "enabled": true,
      "myOption": "custom-value"
    }
  }
}
```

### 2. Implement Your Logic (`src/index.ts`)

The `applyPlugin` function receives the CDS runtime and a matched service instance. Add event handlers, wrap methods, or decorate the service:

```typescript
export function applyPlugin(cds: any, service: any): void {
  service.before('READ', '*', (req: any) => {
    cds.log('plugin').info('Intercepted READ on', req.target?.name);
  });
}
```

### 3. Choose Your Hook (`cds-plugin.js`)

Update the lifecycle hook and service filter in `cds-plugin.js` to match your use case:

```javascript
// For RemoteService plugins:
cds.on('connect', (service) => {
  if (service instanceof cds.RemoteService) {
    applyPlugin(cds, service);
  }
});

// For ApplicationService plugins:
cds.on('serving', (service) => {
  if (service instanceof cds.ApplicationService) {
    applyPlugin(cds, service);
  }
});
```

### 4. Add Utilities (`src/utils.ts`)

Place reusable helpers in `src/utils.ts` and import them from your plugin logic.

## Configuration

Users of your plugin configure it via `cds.env` (package.json or .cdsrc.json):

```json
      "enabled": true

  }
}
```

## Publishing Your Plugin to npm

This template is marked `"private": true` in `package.json` so that the template itself is never accidentally published to npm. The automated release workflow (`release.yml`) will also skip the npm publish step while this flag is set.

When you are ready to publish **your own plugin** derived from this template:

### Manual publish

1. Set your plugin's `name`, `version`, `description`, `author`, and `repository` in `package.json`
2. Remove the `"private": true` line from `package.json`
3. Build and publish:

```bash
npm run build
npm publish
```

The `prepublishOnly` script ensures TypeScript is compiled before every publish.

### Automated publish via GitHub Actions

The included `release.yml` workflow uses [semantic-release](https://semantic-release.gitbook.io/) to publish on every merge to `main`:

1. Complete the manual steps above (remove `"private"`, set package metadata)
2. Create an npm **Automation token** at <https://www.npmjs.com/settings/\<user\>/tokens>
3. Add it as the `NPM_TOKEN` secret in your repo: **Settings → Secrets and variables → Actions**

After that, every merge to `main` that follows [Conventional Commits](https://www.conventionalcommits.org/) will automatically version, tag, and publish the package.
